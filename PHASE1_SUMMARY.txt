───────────────────────────────────────────────────────────────────────────────
DATABASE SCHEMA
───────────────────────────────────────────────────────────────────────────────

Table: browser_sessions
  • Session metadata (id, name, url, active state)
  • Timestamps (created_at, last_activity)
  • Indexes on: is_active, last_activity

Table: browser_logs
  • Unified log storage (console, error, network, DOM, storage)
  • Foreign key to browser_sessions (CASCADE delete)
  • Supports both session_id and port-based logs
  • Indexes on: session_id, port, timestamp, type, level

Table: visual_baselines
  • Screenshot storage for regression testing
  • Foreign key to browser_sessions (CASCADE delete)
  • Unique constraint on (session_id, url, selector)
  • Indexes on: session_id, url, hash

Table: storage_migrations
  • Tracks applied schema versions
  • Ensures idempotent migration application

───────────────────────────────────────────────────────────────────────────────
ARCHITECTURE DECISIONS
───────────────────────────────────────────────────────────────────────────────

1. Storage Pattern:
   • Two-tier: In-memory cache (500 logs) + SQLite persistence
   • Writes go to both layers
   • Reads prefer storage for complete history
   • Balances performance with durability

2. Session Pooling:
   • Reuses browser instances to reduce overhead
   • Automatic cleanup prevents resource leaks
   • Configurable limits prevent memory exhaustion

3. Log Schema:
   • Unified table for all log types (console, error, network)
   • Supports both session-based and port-based logs
   • Flexible schema accommodates future log types

4. Migration Strategy:
   • SQL files in version-numbered order
   • Tracked in storage_migrations table
   • Idempotent (safe to re-run)
   • Forward-only (no rollback)
